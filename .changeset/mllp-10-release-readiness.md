Phase 10 — release readiness for `0.0.1` (MLLP-10).

**No `emit()` reachable from a callback we do not own may go uncontained — in any class (the safety fix).** `EventEmitter.emit()` calls listeners **synchronously**, so a throwing subscriber unwinds the entire stack it was invoked from. On this package's hot paths that stack bottoms out in a socket's `'data'`/`'error'`/`'secureConnect'` listener, a `net.Server`'s `'connection'` listener, a `tls.Server`'s `'tlsClientError'` listener, or the `catch` of a `void`-ed async task — so the throw became an **uncaught exception / unhandled rejection that killed the process**, taking every other connection and every in-flight durable commit with it. A consumer's broken metrics tap could take down an MLLP interface.

The conformance gate refuted this fix **four times**, each round on a route the previous scope had missed:

1. **The decoder's own throw.** `Connection` fed `FrameReader.push(chunk)` with no `try`/`catch`. Reachable on a **default server from one byte**: `SERVER_DEFAULT_FRAMING` leaves `allowMissingLeadingVt` off, so any non-whitespace byte where a `VT` was expected threw `MLLP_MISSING_LEADING_VT` (`MLLP_FRAME_TOO_LARGE` reached the same path). A single stray keepalive character from a real interface engine was enough.
2. **`emit('error')` with no listener** → Node's `ERR_UNHANDLED_ERROR`, raised *from inside the catch block that was the fix for (1)*.
3. **A throwing `'message'`/`'warning'` subscriber** — `onFrame` dispatches synchronously inside `push()`.
4. **The five lifecycle emits** — `destroy()` → `_transition()` → `emit('stateChange'|'close'|…)` runs *inside* the catch block, and a throw raised inside a `catch` is not caught by it.
5. **`MllpServer` and `MllpClient` entirely** — the rule had been scoped to `Connection`, but **the hazard belongs to the call stack, not to a class.** Four more live kills: the server's `'nack'`, `'connection'` and `'tlsClientError'` emits, and the client's `'securityWarning'`.

Two of those are load-bearing well beyond crash-safety:

- **A throwing `'nack'` subscriber suppressed the fail-safe negative ACK.** The emit sat inside the `catch` of `_sendCommitAck` — an `async` method the caller `void`s — *before* `_dispatchAck`. So a broken metrics tap meant the `AE` **was never sent**, leaving the sender waiting on an acknowledgement for a message the server had already failed to commit. This is the exact hazard the commit contract exists to prevent, reached from the observability path.
- **A throwing `'message'` subscriber broke ACK correlation on the client.** It ran before `_onAckPayload`, so the ACK never reached the correlator and `send()` **hung forever**.

Now: `src/internal/safe-emit.ts` (`safeEmit` / `safeEmitError`) is the single containment primitive, used by `Connection._dispatchContained`, `MllpServer._emitContained`, and `MllpClient._emitContained`. **Every `this.emit(` in `src/` sits inside a containment wrapper** (or an inline `try`/`catch`), with exactly one disclosed exception — the deliberate fail-loud accept-loop error forwarder described below.

**ACK serialization is safe against peer- and caller-controlled bytes (a sixth route, and the only peer-triggerable one).** `buildRawAck` decoded the inbound with `ascii`, which masks the high bit (`byte & 0x7f`). That was wrong twice: (1) it violated HL7 v2.5.1 §2.9.2.2 — MSA-2 must echo the inbound MSH-10 **verbatim**, but a control-ID byte `0x8B` silently became `0x0B`, a *different* id, breaking the sender's own ACK correlation for any non-ASCII charset; and (2) `0x8B → 0x0B` is a **VT** and `0x9C → 0x1C` is an **FS**, so `ascii` *manufactured framing delimiters* out of ordinary payload bytes. A peer sending one high-bit byte in an echoed MSH field made the ACK payload contain a real VT/FS, which `encodeFrame` (strict) rejected — and that throw escaped the `void`-ed `_sendCommitAck`, **crashing the whole server on peer-controlled input, with no consumer bug at all**, and suppressing the fail-safe ACK. Fixed: `buildRawAck` uses `latin1` (byte-exact, cannot synthesize a delimiter — a delivered payload cannot itself contain VT/FS), and `_dispatchAck` is now **total** — a frame failure (still reachable via a caller's `autoAck: fn` returning arbitrary bytes) surfaces as a connection `'error'` and the message goes un-ACKed (fail-safe), never as a process kill. New suite `test/server/ack-serialization-safety.test.ts`. A fatal framing error surfaces as a frozen `'error'` (`phase: 'receive'`, `connectionCause: 'framing-fatal'`, the `MllpFramingError` preserved as `cause`) and destroys **that connection only** — the server drops the one bad peer and keeps serving. A fatal framing error is also now reported **exactly once** (`destroy(err)` → `transport.destroy(err)` made the socket echo it back through `_onTransportError`, emitting a second, causeless `'error'`). **One deliberate exception is preserved:** the server's `net.Server` error forwarder still re-emits unguarded when there is **no** `'error'` listener **and** the server is serving — Node's fail-loud convention for accept-loop errors (`EMFILE`/`ENFILE`), because a silent accept outage on a healthcare listener must be impossible.

The existing suites missed all of it because the in-memory transport wraps delivery in `try`/`finally`, re-routing the throw to the *writer*. Only a real socket reproduces it. `test/connection/receive-containment.test.ts` and `test/server/framing-error-containment.test.ts` now attach a throwing subscriber to **every event of all three classes at once**, over real loopback sockets — so a new event emitted uncontained fails the suite.

**`framing-fatal` is classified permanent — no more reconnect storm.** `isTransientConnectionError` switches on `err.code` and fell through to `default: return true`, so a `MllpFramingError` was **transient**: `createStarterClient` (where `autoReconnect` defaults **on**) retried forever against a peer that was not speaking MLLP — an HTTP probe, a health check, a wrong-port misconfiguration — turning the backoff into an unbounded storm against an interface engine. `MLLP_*` codes are now permanent, alongside the TLS classes, for the same reason: every reconnect meets the same bytes. **Public API:** `ConnectionErrorCause` gains `'framing-fatal'`.

**A `'message'` observer can no longer suppress the ACK.** `MllpServer` emits `'message'` to observers *before* ACK dispatch (D-03). A throwing observer aborted the handler before the ACK was sent — so one broken logger silently turned every message into a no-ACK, and every sender resent forever with nothing to diagnose it by. The emit is now contained: the throw surfaces on `'error'` and the commit contract proceeds untouched. The ACK decision belongs to `ServerOptions.onMessage` (the durable-commit step), not to a metrics tap.

**Release-pipeline fix (this was a hard blocker).** The shared `cosyte/.github` release workflow drives Changesets with `version: pnpm run version`, but no `version` script existed — `pnpm run version` failed with `ERR_PNPM_NO_SCRIPT`, so the "Version Packages" PR could never be opened and `0.0.1` could never have been released. Added `"version": "changeset version && node scripts/sync-version.mjs && prettier --write package.json src/index.ts"`.

**`VERSION` export no longer lies about the release.** `VERSION` was hardcoded `"0.0.0"` in `src/index.ts` while `changeset version` bumps only `package.json` — the published `0.0.1` would have exported `"0.0.0"`. The new `scripts/sync-version.mjs` rewrites the constant from `package.json` as part of the `version` script. `test/sanity.test.ts` now compares the export against `package.json` instead of asserting one hardcoded literal against another (the old test would have stayed green through exactly this drift). `VERSION` is also now typed `string` rather than the literal type `"0.0.0"`, which had been leaking the current release into consumers' `.d.ts` and making an equality check against any other version a compile error.

**Publish pipeline proven without burning a version.** Added `publish:dry` (`pnpm publish --dry-run --no-git-checks`). Verified end to end: `prepublishOnly` (clean → typecheck → lint → test → build → `attw`) green; `pnpm run version` consumes the pending changesets to exactly `0.0.1` and syncs the `VERSION` export; the dry-run packs 24 files (`dist/` + README + LICENSE + CHANGELOG) with public access and no `src/`, `test/`, or `vendor/` leakage; `pack:docs` produces both docs artifacts. **No version was published.** Scope note: a dry-run exercises neither OIDC nor npm provenance, so the provenance step itself is *not* covered — and because `publishConfig.provenance` is unconditionally `true`, **the repo must be flipped public before the first publish**, or provenance generation fails. That ordering belongs to the human publish gate.

**Full `docs-content/` transport guide.** New pages — **Framing & tolerance** (wire format, the opt-in tolerance flags with a `FrameReader`-vs-`MllpServer` default table, the stable warning-code registry, what throws and what a throw does, bounded accumulators, the PHI contract on diagnostics), **ACKs & the commit contract** (fail-safe ACK semantics, `autoAck` modes, the transport-accept caveat, correlation, `ack-from-hl7`), **Connection, reconnect & backpressure** (the 6-state machine, jittered backoff, the transient-vs-permanent classifier, keepalive vs dead-peer timeout, high-water marks, and an honest account of shutdown), and **Known limitations & non-goals** (at-least-once at best, no queue/replay, no R2, no Epic/Cerner differential verification, no PKI, not byte-transparent, pre-stable API). Sidebar updated; all examples synthetic.

**Docs accuracy fixes** (found by the conformance gate, which refuted the first cut of these docs):

- `intro.md` called the decoder liberal outright. It is **strict by default** — tolerance is opt-in per flag, and it is `MllpServer` that ships tolerant defaults (`allowFsOnly`, `allowLfAfterFs`, `allowLeadingWhitespace`; `allowMissingLeadingVt` stays off even there).
- `MLLP_TRAILING_BYTES` is **not** "benign junk between frames". It fires on a `VT` appearing *mid-payload* — which **discards the accumulated partial payload**, i.e. a truncated message — and on a stray byte after `FS` under `allowFsOnly`. Documented as something to alert on.
- **`close()` does not drain in-flight messages.** It *rejects* them (`MllpConnectionError({ phase: 'close' })`); no drain hook is wired to the `DRAINING` state, so `drainTimeoutMs` does not bound an in-flight ACK wait on the client. A message in flight at shutdown is an **unknown**, not a failure — the receiver may have committed it. Documented in both the reliability guide and the limitations page, with the "await your sends, then close" pattern.
- The absolute PHI claim ("never echoes message content") is now precise: diagnostics never echo a *run* of content, but `MLLP_MISSING_LEADING_VT`'s single-byte `snippet` is by definition the first byte of unframed content.
