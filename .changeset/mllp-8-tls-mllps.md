TLS / MLLPS hardening (Phase 8): `TlsTransport` over `tls.TLSSocket`, client `tls` option with verification on by default and a loud `allowUnverified` opt-out (`'securityWarning'` + `process.emitWarning` on every `secureConnect`, initial and reconnect), server `tls` option with ATNA ITI-19 mutual-TLS `clientAuth` modes (`NONE`/`WANT`/`MUST`) and a content-free `peerCertificate` summary carrying an `authorized` verification flag (under `WANT` a cert can be present yet unverified), non-crashing `'tlsClientError'` handling, and a TLS 1.2 floor (IHE ATNA ITI-19 BCP195). Typed failure modes: `tls-verify` (certificate verification) and `tls-handshake` (TLS-protocol-shaped only — `ERR_SSL_*`/`EPROTO`/OpenSSL alerts; via the new `isTlsProtocolError` export) are both classified permanent — no auto-reconnect loop against a misconfigured or MITM'd endpoint — while pure TCP blips stay transient. Documented TLS 1.3 semantics honestly (RFC 8446 §4.4.2): `connect()` resolving does not guarantee a `MUST` server accepted the client cert; the rejection surfaces as a typed permanent post-connect error, and ACK correlation remains the delivery guarantee. Also hardens server bind safety: default bind host changes `0.0.0.0` → `127.0.0.1` (BREAKING pre-publish, free before first release), and binding a wildcard host requires `allowWildcardBind: true` — detection normalizes the address, catching `::0`, `0:0:0:0:0:0:0:0`, and `::ffff:0.0.0.0` spellings too, with a one-time `MLLP_BIND_ALL_INTERFACES` security warning when opted in.
