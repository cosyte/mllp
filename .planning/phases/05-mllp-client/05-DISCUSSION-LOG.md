## Area 1 — ACK correlation internals

**Options presented:**
- A. Unified `Map<key, PendingAck>` + ES2015 insertion order + sibling graveyard Map; `pipeline:false` = `maxInFlight=1` guard on same store **(Recommended)**
- B. Two distinct correlator classes (FifoCorrelator + ControlIdCorrelator)
- C. Single ordered list with mode-dispatched `match()`
- D. Two layers (ordering list + lookup Map)

**User selection:** A (recommended).

**Notes:** ioredis (~14k★, ~10 yrs) and redis-py both use this exact unified pattern in production. Project's "short, testable functions" guardrail and `getStats()` JSON-shape requirement push toward one bookkeeping store. CLIENT-17 reconnect walk becomes a one-liner over `Map.values()`.

## Area 2 — Reconnect-time in-flight ACK semantics

**Options presented:**
- 5. Hybrid: controlId resends, FIFO rejects with new cause `'in-flight-orphan'` **(Recommended)**
- 1. Treat in-flight identically to queued
- 2. Reject ALL in-flight with `'in-flight-orphan'`
- 4. Configurable `inFlightOnReconnect` option

**User selection:** 5 (recommended).

**Notes:** Extends CLIENT-17's already-locked FIFO/controlId asymmetry symmetrically across queued and in-flight. `correlateByControlId:true` is itself an explicit opt-in to MSH-10 dedupe — silent resend is consenting behavior. The `'in-flight-orphan'` cause code is a NEW addition to the stable cause-code taxonomy (forward-compatible, but renaming is a future breaking change per project's stable-codes guardrail).

## Area 4 — `retryStrategy` hook contract

**Options presented:**
- 3. Rich `RetryContext` object + Composition A **(Recommended)**
- 2. Inspectable `(attempt, lastError) => number | null` (node-redis style)
- 1. Minimal `(attempt) => number` (ioredis style)

**User selection:** 3 (recommended).

**Notes:** Aligns with project's frozen JSON event-payload precedent. Carries `ctx.signal: AbortSignal` from day one (cheap now, breaking later).
